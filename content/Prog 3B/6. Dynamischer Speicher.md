- in `C++` kann zur Laufzeit Speicher dynamisch allokiert werden und wieder freigegeben werden

## 6.1 Statischer vs. dynamischer Speicher
- in bisherigen Beispielen wurde statischer Speicher verwendet
	-> Speicher wurde zur Kompilierzeit reserviert und konnte zur Laufzeit nicht mehr verändert werden.

- Speicherbedarf: Der Compiler weiß zur Compile-Zeit, wie viel Speicher für Variablen oder Objekte benötigt wird.
- Statische Reservierung: Der benötigte Speicher wird bereits in den Maschinencode eingebaut.
- Variablen in Funktionen:
    - Speicher wird zur Laufzeit auf dem Stack reserviert.
    - Speicher wird nach Verlassen der Funktion automatisch freigegeben.
- Trotz Laufzeitverwaltung: Der Speicher auf dem Stack gilt als statischer Speicher, da der Compiler die Größe und das Layout zur Compile-Zeit festlegt.

```cpp
void foo()
{
    // Speicher für a und v wird statisch auf dem Stack reserviert
    int a = 5;
    Vector2D v;
}

int main()
{
    foo();
}
```

- Dynamische Speicherallokation erlaubt uns zur Laufzeit Speicher zu reservieren und wieder freizugeben
	-> Compiler muss <span style="background:#d4b106">nicht</span> im Vorfeld wissen, wieviel Speicher benötigt wird
	-> nützlich, wenn man zur Laufzeit entscheiden will, wieviel Speicher benötigt wird
	-> oder wenn wir Variablen in einer Funktion anlegen wollen, die über den Funktionsaufruf hinaus bestehen bleiben soll.

## 6.2 Speicher allozieren und freigeben
- Operatoren `new` und `delete`

`new`:
- allokiert Speicher für ein Objekt
- gibt einen Zeiger auf den allkokierten Speicher zurück

`delete`:
- gibt allokierten Speicher wieder frei

```cpp
int* p = new int;   // allokiert Speicher für ein int
*p = 5;             // schreibt 5 in den allokierten Speicher
delete p;           // gibt den allokierten Speicher wieder frei
```

- `new` und `delete` rufen jeweils den Konstruktor bzw. [[3. Klassen#3.5 Destruktoren|Destruktor]] des allokierten Objekts auf.
- Existiert ein Konstruktor mit Parametern, kann man diesen bei `new` auch entsprechend aufrufen

```cpp
#include <iostream>

class Vector2D
{
private:
    int x, y;

public:
    Vector2D(int x, int y) : x(x), y(y) { std::cout << "Konstruktor aufgerufen" << std::endl; }
    ~Vector2D() { std::cout << "Destruktor aufgerufen" << std::endl; }
}

int main()
{
    Vector2D* pV = new Vector2D(1, 2);  // Ausgabe: "Konstruktor aufgerufen"
    delete pV;                          // Ausgabe: "Destruktor aufgerufen"
}
```

## 6.3 Smart Pointer
<u>Problem:</u>
- Speicher muss explizit freigegeben werden
- Falls nicht -> **Memory Leak**
	-> d.h. der allokierte Speicher bleibt auch nach der eigentlichen Verwendung der Variablen reserviert und steht nicht mehr zur Verfügung.
	-> passiert das öfter (z.B. in einer Schleife), so wird der Speicher des Computers immer knapper und das Programm wird abstürzen bzw. der Computer nicht mehr benutzbar sein.

```cpp
int main()
{
    int *p;

    while (true)
    {
        p = new int;
        // Speicher wird nicht freigegeben -> Memory Leak
    }

    delete p;
}
```

> [!note]
> Memory Leaks sind **immer** ein Fehler im Programm und sollten unbedingt vermieden werden.

- treten häufig *versteckt* auf
- Fix? -> **Smart Pointer**

- Smart Pointer sind Klassen, die den Speicher automatisch freigeben, sobald Pointer nicht mehr benötigt wird
- arbeiten mit s.g. *reference counting*
	-> bedeutet, dass der Smart Pointer zählt, wie oft eine Referenz auf das Objekt existiert.
	-> wird der Smart Pointer <span style="background:#d4b106">kopiert</span>, so wird der Zähler <span style="background:#ff4d4f">erhöht</span>
	-> wird der Smart Pointer <span style="background:#d4b106">gelöscht</span>, so wird der Zähler <span style="background:#ff4d4f">verringert</span> 
	-> erreicht Zähler den Wert 0, so wird der allokierte Speicher freigegeben.

*Smart Pointer sind im Header `<memory>`definiert und befinden sich im Namespace`std`*

### 6.3.1 Ownership bzw. wer besitzt das Objekt?
Wenn mehrere Zeiger bzw. Referenzen auf das gleiche Objekt existieren…
Wer ist dann für das Löschen des Objekts verantwortlich?

*Ownership*:
- bedeutet, dass ein Objekt einem bestimmten Besitzer zugeordnet ist
- dieser ist für das Löschen des Objekts verantwortlich
- wird der Besitzer gelöscht, so wird auch das Objekt gelöscht
- Smart Pointer implementieren dies -> stellen den Besitzer eines Objekts dar
- mittels *Move Semantics* kann Besitzer eines Objekts übertragen werden

### 6.3.2 'std::shared_ptr'
- `std::shared_ptr` ist ein Smart Pointer
- implementiert s.g. *shared ownership*
	-> bedeutet, dass ein Objekt mehreren `std::shared_ptr` zugeordnet werden kann
	-> Speicher wird erst freigegeben, wenn der letzte `std::shared_ptr` gelöscht wird.

- wird mittels `std::shared_ptr<T>` deklariert
	-> `T` ist Typ des Objekts, auf das der `std::shared_ptr` zeigt
- im Konstruktor wird Zeiger auf das Objekt übergeben, das der `std::shared_ptr` verwalten soll,
	- z.B. `std::shared_ptr<int> p(new int(5));`
- Auf Inhalt des `std::shared_ptr` greift man mittels des Dereferenzoperators `*` zu
	-> wie bei regulären Zeigern
	-> z.B. `*p = 5;`.

```cpp
#include <memory>
#include <iostream>

int main()
{
    int *pInt = new int(5);

    {
        std::shared_ptr<int> p1(pInt);  // p1 zeigt auf das Objekt pInt

        {
            std::shared_ptr<int> p2 = p1; // p2 zeigt auf das selbe Objekt wie p1
            // p2 wird am Ende des Blocks gelöscht, aber das Objekt bleibt erhalten (da p1 noch existiert)
        }

        std::cout << *p1 << std::endl;  // Ausgabe: 5
        // Nun wird p1 gelöscht. Da p1 der letzte shared_ptr ist, wird auch das Objekt gelöscht.
    }

    // pInt zeigt auf ein bereits gelöschtes Objekt -> undefined behavior
    std::cout << *pInt << std::endl;

    return 0;
}
```

- `std::share_ptr` einfachste zu verwendete Art von Smartpointern
- Aber *shared ownership* nicht immer das was man benötigt
- Falsche Verwendung kann dazu führen, dass Objekte nicht freigegeben werden

> [!note]
> Vermeiden Sie es, `std::shared_ptr` zu verwenden, wenn Sie nicht wirklich _shared ownership_ benötigen.

### 6.3.3 'std::unique_ptr'
- ein Smart Pointer, der s.g. *unique ownership* implementiert
	-> Ein Objekt kann nur einem `std::unique::ptr` zugeordnet werden
		- wird `std::unique::ptr` gelöscht, wird auch das Objekt gelöscht
		- Besitz ist eindeutig geregelt

- als `std::unique_ptr<T>` deklariert

- soll das Objekt einem anderen `std::unqiue_ptr` zugeordnet werden -> Objekt mit *Move Semantics* übertragen
	- `std::move`, z.B. `std::unique_ptr<int> p2 = std::move(p1);`


```cpp
#include <memory>
#include <iostream>

int main()
{
    std::unique_ptr<int> p1(new int(5));

    std::unique_ptr<int> p2;

    p2 = p1; // Compilerfehler: unique_ptr kann nicht kopiert werden (dies würde zu "shared ownership" führen)

    p2 = std::move(p1); // p2 zeigt nun auf das Objekt, das zuvor p1 zugeordnet war
                        // p1 verliert das Objekt und zeigt nun auf nullptr
    std::cout << *p2 << std::endl; // Ausgabe: 5

    p2.reset(new int(10)); // p2 zeigt nun auf ein neues Objekt, das alte Objekt wird gelöscht
    std::cout << *p2 << std::endl; // Ausgabe: 10

    return 0;
}
```

Ein häufiges Pattern bei der Verwendung von `std::unique_ptr` ist, dass wir auf ein von einem unique_ptr verwaltetes Objekt zugreifen wollen, ohne das Objekt zu übernehmen (z.B. in einer Funktion). Dazu können wir eine const-Referenz auf das Objekt übergeben, z.B.:

```cpp
#include <memory>
#include <iostream>

void foo(const std::unique_ptr<int>& p)
{
    std::cout << *p << std::endl;
}

int main()
{
    std::unique_ptr<int> p(new int(5));
    foo(p);

    return 0;
}
```

> [!note]
> In den meisten Anwendungsfällen ist _unique ownership_ und damit `std::unique_ptr` die beste Methode.

### 6.3.4 'std::make_unique' und 'std::make_shared'
- vereinfacht Erstellung von `std::unique_ptr` und `std::shared_ptr`
- Dabei werden die Parameter an den Konstruktor des Objekts weitergegeben.

Anstatt <br>`std::unique_ptr<Vector2D> p(new Vector2D(1.,2.));` zu schreiben,<br>-> können wir auch <br>`std::unique_ptr<Vector2D> p = std::make_unique<Vector2D>(1.,2.);` verwenden.

Noch einfacher mit `auto`:
- Typ muss nicht explizit angegeben werden
- `auto p = std::make_unique<Vector2D>(1.,2.);`

> [!note]
> Nutzen Sie `auto` nur dann, wenn der Typ des Pointers klar ist und der Code dadurch lesbarer wird. Vermeiden Sie `auto` bei trivialen Typen wie `int`, `float`, etc.

### 6.3.5 'std::dynamic_pointer_cast' und 'std::static_pointer_cast'

- um vom `std::shared_ptr` auf einen anderen Typ zu casten
- wie `C++`-Casts, aber mit `std::shared_ptr`

```cpp
#include <memory>

class A {
public:
    virtual void foo(){};
};
class B : public A {};

int main()
{
    std::shared_ptr<A> pA = std::make_shared<A>();
    std::shared_ptr<B> pB = std::dynamic_pointer_cast<B>(pA);

    return 0;
}
```
