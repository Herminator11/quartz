## 4.1 Eigenschaften von Klassen
Bevor man sich für ein gegebenes Problem ein Programm entwickeln, sollte man sich überlegen, wie man dies in Klassen abbilden kann.

Stellen Sie sich vor, Sie sollen für ein 2D-Spiel eine Klasse implementieren, die ein Spielobjekt repräsentiert, welches sich auf dem Bildschirm bewegen kann. 
Nennen wir diese Klasse `GameObject`. 
Wir können die Anforderungen bzw. Eigenschaften von `GameObject` zuerst einmal umgangssprachlich definieren:

Ein `GameObject`:
- hat eine *Position* auf dem Bildschirm.
- kann sich auf dem Bildschirm bewegen.
- kann auf dem Bildschirm gezeichnet werden.

-> Positionsverwaltung vom Objekt
- weil wir ein 2D-Spiel machen, können wir die Position als Koordinatenpaar `(x, y)` darstellen.
- also ein zweidimensionaler Vektor `Vector2D`

Ein `Vector2D`:
- hat eine x- und eine y-Koordinate.
- kann mit einem anderen Vektor addiert werden.
- kann mit einem Skalar multipliziert werden.

Diese Anforderungen kann man nun in Klassenattributen und -methoden überführen.
Dafür machen wir ein s.g. *UML-Klassendiagramm*.
Ein Klassendiagramm besteht aus Klassen mit Attributen und Methoden.
![](static/assets/progb-1.png)
Man sieht die Attribute der Klassen (jeweils im oberen Bereich) als auch die Methoden (im unteren Bereich), inklusive ihrer Signatur.
- Zusätzlich sind die Sichtbarkeiten der Attribute und Methoden angegeben (`+` für *public*, `-` für *private*).
- Außerdem kann man die Beziehungen zwischen den Klassen darstellen
	-> hier hat `GameObject` eine *Aggregation* zu einem `Vector2D`, dargestellt durch die Linie mit Diamant am Anfang.
	=> bedeutet, dass ein `GameObject` ein `Vector2D` hat bzw. *besitzt*.

## 4.3 Vererbung
- bedeutet, dass eine Klasse von einer anderen Klasse *erbt*.
- Erbenden Klasse übernimmt dabei die Attribute und Methoden der Basisklasse.

Nehmen wir an, wir wollen ein weiteres Spielobjekt implementieren, das den Spieler repräsentiert. Auch dieses Objekt hat eine Position, kann sich bewegen und auf dem Bildschirm gezeichnet werden. Über Vererbung können wir die Klasse `Player` von der Klasse `GameObject` ableiten.

![](static/assets/progb-2.png)
- Beziehung `Player` *erbt von* `GameObject` wird durch die Linie mit Pfeil dargestellt.

Im Code wird Vererbung bei der Deklaration der Kindklasse durch einen Doppelpunkt und den Namen der Elternklasse angegeben:
`class <Kindklasse> : public <Elternklasse> {}`.

Das `public` steht für die Sichtbarkeit der geerbten Attribute und Methoden.
-> bei `public` behalten wir die Sichtbarkeit der geerbten Elemente bei
	=> d.h. alles, was in der Elternklasse `public` ist, ist es auch in der Kindklasse).
	=> `private` würde bedeuten, dass alle geerbten Elemente in der Kindklasse `private` sind und nicht von außen zugreifbar.

Vollständige Implementierung der Klasse:

```cpp
class Player : public GameObject {
    private:
        int health;
    public:
        Player() : GameObject(), health(100) {}
        Player(Vector2D position) : GameObject(position), health(100) {}
        void printHealth() const {
            std::cout << "Health: " << health << std::endl;
        }
};
```
- in den Konstruktoren der Kindklasse muss der Konstruktor der Elternklasse explizit aufgerufen werden
	-> dies geschieht mit dem `:`-Operator und dem Namen der Elternklasse nach der Deklaration des Konstruktors.
	-> Bei Initialisierungsliste: "Wir rufen einfach die Konstruktoren der jeweiligen Attribute auf."

### 4.3.1 Auswirkungen von Vererbung
Kindklasse erbt Attribute und Methoden der Elternklasse.
-> Kindklasse verfügt über alle Attribute und Methoden der Elternklasse

Beispielsweise können wir in der Klasse `Player` auf die Methode `draw` zugreifen, obwohl sie in der Klasse `GameObject` definiert ist:

```cpp
Player player;
player.draw(); // ruft eigentlich GameObject::draw() auf
```

> [!info]
> Vererbung kann auch in einer Kette von mehreren Klassen angewendet werden, d.h. eine Klasse `A` kann von einer Klasse `B` erben, die wiederum von einer Klasse `C` erbt. Außerdem kann in `C++` eine Klasse von mehreren Klassen gleichzeitig erben.

### 4.3.2 Sichtbarkeiten bei Vererbung
In [[3. Klassen#3.2 Sichtbarkeit und Zugriff|Kapitel 3.2]] haben wir die Sichtbarkeiten `public` und `private` kennengelernt, die den Zugriff auf Attribute und Methoden einer Klasse regeln.
Bei Vererbung gibt es zusätzlich die Sichtbarkeit `protected`. 
	-> besagt, dass ein Attribut oder eine Methode nur in der Klasse selbst und in abgeleiteten Klassen sichtbar ist.

Um Beispielsweise in `Player` direkt auf die Position des Spielers zuzugreifen, müssten wir die Sichtbarkeit von `position` in der Elternklasse auf `protected` setzen:

```cpp
class GameObject {
    protected:
        Vector2D position;
    public:
        ...
};
```
> [!note]
> Die Sichtbarkeiten in C++ sind:
> 
> - `public`: Zugriff von außen.
> - `protected`: Zugriff von in der Klasse selbst und in abgeleiteten Klassen.
> - `private`: Zugriff nur in der Klasse selbst.
> 
> Nutzen Sie immer die restriktivste Sichtbarkeit. Auch bei Vererbung reicht in den vielen Fällen `private` aus.

### 4.3.3 Virtuelle Methoden und Polymorphie

Virtuelle Methoden:
- weitere Eigenschaft von Vererbung
- sind Methoden, die in der Elternklasse definiert sind und in der Kindklasse überschrieben werden können.
=> ermöglicht es, dass die Kindklasse eine eigene Implementierung der Methode hat. Dieser Mechanismus wird auch als **Polymorphie** bezeichnet.

- um eine Methode virtuell zu machen, fügt man das Wort `virtual` vor den Methodennamen in der Elternklasse hinzu.
- In der Kindklasse kann man die Methode dann *überschreiben*
	-> d.h. wir definieren eine Methode mit der gleichen Signatur, die nun bei Objekten des Typs der Kindklasse anstatt der gleichnamigen Funktion der Elternklasse aufgerufen wird.

Nehmen wir an, wir wollen die Methode `move` in der Klasse `GameObject` virtuell machen, damit in der Klasse `Player` eine eigene Implementierung möglich ist, die das Spielerobjekt in Abhängigkeit von der Tastatureingabe bewegt.

```cpp
class GameObject {
    ...
    public:
        virtual void move() {
            // Standardimplementierung
        }
    ...
};
```
In der Klasse `Player` können wir die Methode dann überschreiben:
```cpp
class Player : public GameObject {
    ...
    public:
        void move() override {
            // Spielerbewegung
        }
    ...
};
```

> [!info]
> **Polymorphie** bedeutet, dass ein Objekt einer Kindklasse als Objekt der Elternklasse verwendet werden kann, wobei aber die spezifische Implementierung der Kindklasse verwendet wird. Dies ermöglicht es, verschiedene Implementierungen der gleichen Schnittstelle zu verwenden.

- wird `move` aufgerufen, wird je nach konkretem Objekttyp die entsprechende Implementierung ausgeführt (auch wenn der Zeiger auf das Objekt vom Typ der Elternklasse ist).
- Das folgende Beispiel verdeutlich dies:

```cpp
Player player;
GameObject *obj = (GameObject*) &player; // cast zu GameObject*
obj->move(); // ruft Player::move() auf
```

#### 4.3.3.1 Verdecken von Methoden
- eine bestimmte Methode in der Vererbungshierarchie wird nur dann überschrieben, wenn sie in mindestens einer der Kindklassen als `virtual` deklariert ist.
- Ist dies nicht der Fall, "verdeckt" eine Methode in der Kindklasse die Methode der Elternklasse, statt sie zu überschreiben.
- Um sicherzustellen, dass eine Methode in einer Kindklasse eine virtuelle Methode der Elternklasse überschreibt, können wir sie optional mit dem Schlüsselwort `override` markieren.

In diesem Beispiel wird ein Compilerfehler erzeugt, wenn die Methode in der Elternklasse nicht virtuell ist:

```cpp
class GameObject {
    ...
    public:
        void move() {
            // Achtung: Methode nicht virtuell!!
        }
    ...
};

class Player : public GameObject {
    ...
    public:
        virtual void move() {
            // Methode verdeckt GameObject::move()
        }
    ...
};
```
Im Beispiel unten wird nun GameObject::move() aufgerufen, da die Methode in GameObject nicht virtuell ist:

```cpp
Player player;
GameObject *obj = (GameObject*) &player; // cast zu GameObject*
obj->move(); // ruft GameObject::move() auf, da diese Methode nicht virtuell ist
```

Markieren wir die Methode in Player als `override`, teilen dem Compiler also mit, dass wir hier eigentlich etwas überschreiben möchten, erhalten wir einen Compilerfehler:

```cpp
class Player : public GameObject {
    ...
    public:
        void move() override {
    // Compilerfehler: Methode überschreibt keine virtuelle Methode
        }
    ...
};
```
### 4.3.4 Typecasting
Im Beispiel oben haben wir einen Zeigen vom Typ `Player` auf einen Zeiger vom Typ `GameObject` gecastet. Wir haben dazu die klassische `C`-Syntax verwendet, wobei wir den Zieltypen in Klammern vor den Ausdruck gesetzt haben:
`GameObject*) &player`.
-> weiterhin gültig, aber gibt bessere Möglichkeiten in `C++`

Das Equivalent zu diesem Ausdruck ist der **static_cast**:
`GameObject *obj = static_cast<GameObject*> (&player);`, wobei der Zieltyp in spitzen Klammern angegeben wird.

- static_cast führt keine Überprüfung des Typs durch, d.h. es wird einfach angenommen, dass der Zeiger korrekt gecastet werden kann.
- wird häufig für primitive Datentypen verwendet, um z.B. von einem `int` zu einem `double` zu casten

> [!note]
> Nutzen Sie den _static_cast_ nur, wenn Sie sicher sind, dass der Zeiger korrekt gecastet werden kann oder für primitive Datentypen.

- Im Kontext von Vererbung und Polymorphie ist es sinnvoll den **dynamic_cast** zu verwenden.
- dynamic_cast führt eine Überprüfung des Typs durch und gibt im Fehlerfall `nullptr` zurück.
- Nützlich, wenn man sich nicht sicher ist, ob der Zeiger tatsächlich auf ein Objekt der Zielklasse zeigt.
- *dynamic_cast* wird wie folgt verwendet:
`GameObject *obj = dynamic_cast<GameObject*>(&player);`.

```cpp
class GameObject {...};
class Player : public GameObject {...};
class Enemy : public GameObject {...};

void isPlayer(GameObject *obj) {
    Player *player = dynamic_cast<Player*>(obj);
    if(player) {
        std::cout << "Objekt ist vom Typ Player" << std::endl;
    }
    else {
        std::cout << "Objekt ist nicht vom Typ Player" << std::endl;
    }
}

int main() {
    Player player;
    Enemy enemy;
    isPlayer(&player); // Ausgabe: "Objekt ist vom Typ Player"
    isPlayer(&enemy); // Ausgabe: "Objekt ist nicht vom Typ Player"
}
```

> [!info]
> Der _dynamic_cast_ ist sicherer als der _static_cast_, da er eine Überprüfung des Typs durchführt.

### 4.3.5 Abstrakte Klassen
- Eine Klasse, die mindestens eine *rein virtuelle* Methode enthält
- eine *rein virtuelle* Methode ist eine Methode, die in der Elternklasse deklariert, aber nicht definiert ist.
- Sie hat die Form `virtual void methode() = 0;`.

- können nicht instanziiert werden,
	-> d.h. es können keine Objekte von ihnen erzeugt werden.
	- dienen als Vorlage für konkrete Klassen, die von ihnen erben.
	- können aber als Zeiger oder Referenzen verwendet werden.
	- In Praxis oft als *Interfaces* verwendet, d.h. sie beschreiben eine Schnittstelle, die von konkreten Klassen implementiert werden muss.

## 4.4 Modultechnik
In den Beispielen wurde die Klasse gleichzeitig *deklariert* und *definiert*.
Sinnvoll ist es jedoch, die Deklaration und Definition von Klassen zu trennen.
Die Deklaration erfolgt in einem <span style="background:#d4b106">Header</span>-File (`.h` oder `.hpp`), die Definition in einer <span style="background:#d4b106">Quelltextdatei</span> (`cpp`).

Deklaration enthält
- sämtliche Attribute
- Deklaration der Methoden
- **nicht** deren Implementierung
Bei unserer Klasse `Player` könnte die Deklaration in einer Header-Datei `Player.h`so aussehen:

Player.h
```cpp
#pragma once
#include "GameObject.h"

class Player : public GameObject {
    private:
        int health;
    public:
        Player();
        Player(Vector2D position);
        void printHealth() const;
};
```
- `#pragma once`verhindert, dass die Header-Datei mehrfach eingebunden wird.

- Methoden werden separat in einer `.cpp`-Datei implementiert.
- Methodenname besteht aus Klassennamen gefolgt von `::`und dem Methodennamen.

In unserem Beispiel könnte die Datei `Player.cpp` wie folgt aussehen:
Player.cpp
```cpp
#include "Player.h"

Player::Player() : GameObject(), health(100) {}

Player::Player(Vector2D position) : GameObject(position), health(100) {}

void Player::printHealth() const {
    std::cout << "Health: " << health << std::endl;
}
```
