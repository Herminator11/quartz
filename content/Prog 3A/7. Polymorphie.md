# Roll Out
- Prozess, bei dem Software in Produktionsumgebung überführt und für Endbenutzer verfügbar gemacht wird.
- Fokus auf reibungslose Bereitstellung -> Fehler minimieren

- Phasen des Roll-Outs:
1. Planung: Definition von Zielen, Zeitplänen und Teams
2. Testphase: Sicherstellen, dass die Software stabil ist
3. Deployment: Installation in der Aktivierung in der Produktionsumgebung
4. Monitoring: Überwachung nach dem Roll-Out, um Probleme schnell zu beheben

## Transport
- Transport beschreibt den Prozess des *Deployments* der Software von der Entwicklungsumgebung bis zum Endbenutzer

## Diversität der Hardware und Software
- sehr schwierig eine einheitliche Plattform für alle Benutzer zu gewährleisten
- Beispiele:
	- Hardware (unterschiedliche CPU-Architekturen, RAM-Größen, GPU-Fähigkeiten)
	- Software (versch. Betriebssysteme, Versionen)
- Lösungen:
	- Verwendung standardisierter Entwicklungsumgebungen
	- Virtualisierung und Containerisierung

## Virtualisierung
- mehrere VMs auf einer physischen Maschine betreiben
- ermöglicht es, einen "idealtypischen Rechner" durch verschiedene Host-Systeme zu simulieren

-> gleiche Software läuft in allen Umgebungen identisch
	- egal ob Windows, Linux, macOS, alle können denselben Rechner simulieren
-> Probleme, die auf unterschiedlicher Hardware auftreten könnten, werden vermieden

## Containerisierung
- Alternative zur Virtualisierung
- Container bieten isolierte Umgebung für Anwendungen (ähnlich wie VMs, aber ohne eigenes OS)
- nutzen den gleichen OS-Kernel
	-> Voraussetzung ist eine gemeinsame Betriebsfamilie (z.B. Unix)

Container-Image ist wie ein Bauplan.
- enthält alle notwendigen Komponenten, damit eine Anwendung funktioniert
- Container-Images sind klein und standardisiert -> können problemlos transportiert werden
- Beispiel: Ein Container der auf einem Entwickler-Laptop erstellt wurde, kann unverändert auf einem Produktionsserver gestartet werden

### **Vergleich zu virtuellen Maschinen**

|**Merkmal**|**Virtuelle Maschine (VM)**|**Container**|
|---|---|---|
|Betriebssystem|Eigene Gastbetriebssysteme pro VM|Gemeinsamer Kernel mit Host|
|Speicherbedarf|Groß (enthält OS + Anwendung)|Klein (nur Anwendung + Libraries)|
|Startzeit|Minuten|Sekunden|
|Transportierbarkeit|Abhängig vom Hypervisor|Plattformübergreifend, solange OS-Kernel kompatibel ist|
|Ressourcenverbrauch|Höher|Geringer|

### Liskovsche Substitutionsprinzip
> [!quote]
> Wenn **S** eine Subklasse von **T** ist, dann sollte sich jedes Objekt von Typ **T** durch ein Objekt von Typ **S** ersetzen lassen, ohne dass die Korrektheit des Programms beeinträchtigt wird.

Anders gesagt:
Eine Subklasse muss sich so verhalten wie die Superklasse, damit man sie ohne Probleme austauschen kann.

> [!note]- Beispiel: Fahrzeuge
> Stell dir vor, du hast ein Programm, das Fahrzeuge verwaltet:
> 1. Superklasse: Fahrzeug
> - Ein Fahrzeug kann starten und stoppen.
> 2. Subklassen: Auto und Fahrrad
> - Ein Auto und ein Fahrrad können ebenfalls starten und stoppen.
> - Beide verhalten sich wie ein allgemeines Fahrzeug.
> <br> Hier kann ein Auto oder ein Fahrrad überall dort verwendet werden, wo ein Fahrzeug erwartet wird, ohne dass das Programm verändert werden muss. Beispiel:
> - Wenn du eine Funktion hast, die ein Fahrzeug startet (`start()`), funktioniert sie sowohl mit einem Auto als auch mit einem Fahrrad.
> <br> Verletzung des LSP
> Jetzt erweitern wir das Beispiel um eine Subklasse "Boot", die auch ein Fahrzeug sein soll.
> - Das Boot hat jedoch eine zusätzliche Einschränkung: Es kann nur starten, wenn es im Wasser ist
> - Wenn du ein Boot in einer Funktion verwendest, die erwartet, dass alle Fahrzeuge immer starten können, führt das zu Fehlern.

- Wenn Subklassen das Verhalten der Superklasse ändern, wird Code schwieriger wiederverwendbar und instabil.
- Funktionen, die für die Superklasse geschrieben wurde, funktionieren nicht mehr richtig mit allen Subklassen.

> [!tldr]
> Das Liskovsche Substitutionsprinzip sorgt dafür, dass der Code stabil und zuverlässig bleibt. Sunklassen sollten die Eigenschaft und das Verhalten der Superklasse nicht einschränken oder verändern, sondern **nur erweitern**. Dadurch bleibt der Code leicht verständlich und flexibel.

## Upcast
![](static/assets/proga-22.png)
- Die Zuweisung des "falschen" Typs an eine Referenz der Oberklasse wird vom Compiler akzeptiert. Man spricht von einer impliziten Typumwandlungm einem **Upcast**.

## Downcast
- erlaubt eine **allgemeine Referenz** (z.B. eine Referenz auf eine Oberklasse) in eine **spezifische Referenz** (z.B. auf eine Unterklasse) umzuwandeln
- -> um auf zusätzliche Eigenschaften oder Methoden der Unterklasse zuzugreifen.

1. Was ist die Situation?
	- Du hast eine Referenz auf eine **Oberklasse** (z.B. Object), aber du weißt, dass das Objekt auf das sie zeigt, eigentlich eine **Unterklasse** ist (z.B. Exception).
2. Warum willst du das tun?
	- Die allgemeine Referenz erlaubt nur Zugriff auf Eigenschaften und Methoden der Oberklasse
	- um auf spezielle Methoden oder Eigenschaften der Unterklasse zuzugreifen, musst dud die Referenz in die Unterklasse "herunterbrechen".
3. Wie funktioniert der Downcast?
	- Referenz so umwandeln, dass auf spezifische "Fähigkeiten" der Unterklasse zugreifen kann
4. Was passiert, wenn der Typ nicht passt?
	- wenn du einen Downcast auf einen falschen typ machst, wirft Java eine`ClassCastException`. 
	- Um das zu vermeiden, prüfst du vorher mit `instanceof`, ob das Objekt wirklich den gewünschten Typ hat.


```java
public void printInfo(Object o) {
    // Prüft, ob o vom Typ Exception ist (oder eine Unterklasse von Exception)
    if (o instanceof Exception) {
        // Führt den Downcast durch, weil wir wissen, dass es sicher ist
        Exception e = (Exception) o;
        // Greift auf die spezielle Methode der Klasse Exception zu
        System.out.println(e.getMessage());
        return;
    }
    // Wenn es keine Exception ist, wird die allgemeine Methode von Object aufgerufen
    System.out.println(o.toString());
}
```


> [!tldr]
> Downcast wird verwendet, wenn du eine allgemeine Referenz (z.B. auf eine Oberklasse) in eine spezifische Referenz (z.B. auf eine Unterklasse) umwandeln möchtest, um auf zusätzliche Eigenschaften oder Methoden der Unterklasse zuzugreifen.

### Übersicht
