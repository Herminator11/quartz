- Software soll Anforderungen entsprechen
	-> möglichst fehlerfrei und effizienz
	-> Anforderungen -> Erfüllungsgrad

Dafür gibt es zwei Ansätze:
## 1. Konstruktive Qualitätssicherung
- von Anfang an soll gute Qualität sichergestellt werden
- Ziel ist es, dass gar nicht erst Fehler entstehen
	- Wie? -> Indem man während der Entwicklung gezielt vorgeht und gute Praktiken anwendet.
- Beispiele:
	- Schulungen
	- Checklisten
	- Vorgehensmodelle
	- Clean Code
	- Pair Programming

## 2. Analytische Qualitätssicherung
- hier wird überprüft, wie gut die Software am Ende wirklich ist.
- man misst, ob Software die Anforderungen erfüllt
	- ob sie effizient arbeitet
	- ob sie fehlerfrei ist
- Beispiele:
	- Reviews
	- Audits
	- Code-Inspektionen
	- Statische Codeanalyse
	- Dynamische Codeanalyse
	- Softwaretests

### Statische Codeanalyse
- Untersuchung des Quellcodes ohne ihn auszuführen
- Dabei werden Metriken berechnet, um den Zustand und die Qualität des Codes zu bewerten.

**Beispiele für Metriken:**
- Anzahl der Codezeilen (SLOC)
	-> *viele Zeilen Code kann auf Komplexität hinweisen (aber auch: wenig ≠ besser)*
- Anzahl der Unterklassen einer Klasse (NOC)
	-> *viele Unterklassen kann bedeuten, dass eine Klasse schlecht strukturiert ist*
- Halstead Länge
	- zählt die Anzahl von Operatoren(z. B. `+`, `-`, `if`) und Operanden (Variablen, Konstanten) in einem Codestück
	-> *viele Operatoren & Operanden bedeuten, dass Code zu verstehen sein könnte*

Durch Kombination solcher Metriken können aussagekräftige *Kennzahlen* entstehen.
 z.B. Halstead-Länge im Verhältnis zu SLOC:
 - Zeigt, ob Code effizient oder unnötig komplex geschrieben ist.

Man kann "Korridore" festlegen, in dem sich diese Kennzahlen bewegen dürfen.

### Dynamische Codeanalyse
#### Profiler
- Profiler ermitteln bspw. Anzahl der Aufrufe einer bestimmten Methode oder die Rechenzeit, die für diese Methode aufgewendet wird.

#### Black-Box-Test
- prüft Software, ohne den Quellcode anzusehen (nicht wie bei White-Box-Test)
- Fokus liegt auf dem Verhalten der Software und es wird über ihre öffentlichen Schnittstellen getestet.

<u>Wie funktioniert ein Black-Box-Test?</u>
Einfacher Fall:
1. Tester gibt Daten ein
2. Software verarbeitet die Eingaben
3. Eingabe und erwartete Ausgabe wird geprüft
	-> Ausgabe wird auch *Testfall* genannt

> [!note]- Beispiel
> **Beispiel:**
> - Du testest einen Taschenrechner.
> - Eingabe: `2 + 2`.
> - Erwartete Ausgabe: `4`.
> - Wenn die Software tatsächlich `4` liefert, besteht sie den Test.

<u>Herausforderugen</u>:
- Schwierigkeit ist, dass nicht alle *möglichen* Eingaben getestet werden können
Lösung -> Testfälle reduzieren:
- Testen von Grenzwerten und Ausnahmesituationen
- Äquivalenzklassentests:
	- man gruppiert ähnliche Eingaben, die das gleiche Ergebnis liefer.
	- Beispiel: *Eine Funktion, die alle Zahlen über 100 als "hoch" und darunter als "niedrig" einstuft. Statt jede Zahl zu testen, prüft man nur einen Wert pro Gruppe: z.B. 99 (für niedrig) und 101 (für hoch).*

### Tests zusammengefasst
**Anforderung**
- Test soll zeitnah zur Entwicklung erfolgen
- Test soll wiederholt bei jeder Änderung durchgeführt werden

**Rahmenbedingung**
- Sehr große Anzahl von Testfällen
- Hoher Zeitaufwand für die manuellen Tests
- Stupide und fehleranfällige Tätigkeit

-> Können Black-Box-Tests automatisiert werden?

### Automatisierter Softwaretest
- Testfälle werden programmiert (<span style="background:#40a9ff">Testcode</span>) und in einer <span style="background:#40a9ff">Testumgebung</span> ausgeführt.
- Tests können so sehr einfach wiederholt werden, sind schnell und kostengünstig.
- Außerdem stellen die automatisierten Testfälle eine implizite Dokumentation der Unit under Test dar.

> [!quote] Grundprobleme von a posteriori Maßnahmen
> Man kann keine Qualität in Software hineinprüfen. Qualität kann man nur in Software hineinkonstruieren.
> 
> Test können Fehler nachweisen, nicht aber die Fehlerfreiheit.

## Test Driven Development (TDD)
Idee: Wir schreiben erst den Testcode, und erst anschließend den Produktivcode

### Die drei Gesetze von TDD
1. Produktionscode darf erst geschrieben werden, wenn es einen scheiternder Test gibt.
2. Test darf nicht mehr Code enthalten, als für das Scheitern und ein korrektes Kompilieren des Tests erforderlich ist.
3. Man darf nur so viel Produktionscode schreiben, wie für das Bestehen des gegenwärtig scheiternden Tests ausreicht.

### Red-Green-Refactor - Zyklus

<span style="background:#ff4d4f">Red</span>
- Ziel: Test schreiben und erwarten, dass er fehlschlägt, da die Funktionalität noch nicht implementiert ist.
- Kein Produktivcode -> Nicht den Code für Funktionalität implementieren, bevor der Test geschrieben und ausgeführt wurde

<span style="background:rgba(205, 244, 105, 0.55)">Green</span>
- Ziel: Test zum Bestehen bringen, indem man minimalen Code schreibt, der erforderlich ist, damit der Test erfolgreich ist.
- nur minimale Änderungen am Testcode / nicht unnötige Änderungen am Testcode vornehmen
	-> Test soll nur überprüft und nicht geändert werden

<span style="background:#40a9ff">Refactor</span>
- Ziel: Code verbessern, ohne das Verhalten zu verändern
	-> leserlicher, effizienter ohne dass Tests fehlschlagen
- keine zusätzliche Funktionalitäten

## Mocks und Stubs
- kommen in automatisierten Tests zum Einsatz
- die zu testende Komponente (die "Unit under Test", UUT) wird dabei isoliert
- Dadurch konzentriert sich der Test auf die Funktionalität der UUT (ohne Auftreten unerwünschter Nebeneffekte)

### Mocking
- Mock ist ein simuliertes Objekt, das die gleiche Schnittstele wie das echte Objekt implementiert.
- wird verwendet, um das Verhalten von Objekten zu simulieren, mit denen die UUT interagiert
- *Die Mocks isolieren das zu testende Modul von den echten Abhängigkeiten.*

> [!example] Beispiel
> Angenommen, wir testen eine Funktion, die mit einer Datenbank kommuniziert, aber wir möchten nicht wirklich eine Datenbankabfrage ausführen.
> -> Stattdessen benutzen wir ein Mock-Objekt, das die gleiche Schnittstelle wie die echte Datenbank implementiert, aber stattdessen nur vorgefertigte Antworten zurückgibt.

<u>Voraussetzung</u>:
- Objekt, was man mocken will, muss eine Schnittstelle haben
- Das echte Objekt und der Mock implementieren diese Schnittstelle, damit das Testobjekt mit beiden kommunizieren kann.

### Stubbing
- Stub ist eine spezielle Art von Mock
- wird verwendet um das Verhalten eines Objekts zu simulieren
- gibt vorab festgelegte Werte zurück (da vor der Nutzung "Regeln" hinterlegt werden)
- diese Werte werden später bei Aufrufen im Rahmen des Tests zurückgeliefert

<u>Warum?</u>
- wenn man sicherstellen möchte, dass bestimmte Methoden nur mit gewünschten Werten ausgeführt werden sollen (keine echte Logik).

> [!example] Beispiel
> Angenommen, wir testen eine Methode, die mit einer API kommuniziert, aber wir möchten vermeiden, dass bei jedem Test eine echte HTTP-Anfrage gesendet wird. 
> -> Wir verwenden einen Stub, der die HTP-Anfrage simuliert und eine feste Antwort zurückgibt, die wir für den Test benötigen.

=> Wir können also mit Mocks und Stubs spezifische Szenarien isoliert testen, ohne sich um externe Faktoren oder unerwünschte Nebeneffekte zu kümmern.
