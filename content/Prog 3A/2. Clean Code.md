## Wiederholung

Was ist static?
```java
public static void main(String[] args) {

}
```

- static -> Methode gehört zur Klasse (nicht zu einem Objekt!)
	-> `static`-Methode kann verwendet werden, ohne vorher ein Objekt der Klasse zu erstellen (man braucht keine Objekte!).

- warum kann main static sein??
	-> `main` ist Einstiegspunkt für das Programm.
	-> wird vom Java-System direkt aufgerufen, ohne dass ein Objekt der Klasse existiert
	-> deshalb muss sie statisch sein, damit sie ohne Objekte funktioniert

```java
public static void main (String [] args) {
 abc();
}

public void abc() { ... }
```
-> funktioniert nicht
- es wurde noch kein Objekt angelegt und somit kann die Methode nicht aufgerufen werden
- `abc` ist keine statische Methode. Sie gehört zu einem Objekt der Klasse und nicht nur Klasse selbst.
- um auf `abc()` zuzugreifen, braucht man ein Objekt der Klasse

Geht das?
```java
public static void main (String [] args) {
	Zahlenfilter z = new Zahlenfilter(); // Objekt wird erstellt
	z.xyz(); // funktioniert
}

public void abc() { ... }

public void xyz() {
	abc(); // funktioniert weil xyz Teil des gleichen Objekts ist
}
```
Wir brauchen immer ein Objekt um eine nicht statische Methode aufzurufen!
- `xyz()` ruft die Methode `abc()` auf, aber das geht, weil beide nicht statisch sind und vom gleichen Objekt verwendet werden.

```java
public static void main (String [] args) {
	Zahlenfilter z = new Zahlenfilter();
}

public void abc() {
	System.out.println("Hello");
}
```
System.out.println:
- `System` -> Klasse von java.lang
- `out` -> statische Klassenvariable / Ausgabestream
- `println` -> Methode von `out`

## Clean Code

Was können wir tun um menschenverständlichen Code zu schreiben:

### Aussagekräftige Namen

```JS
function d(a){
	c=a.length, i=0, s=0;
	while(i<c)s+=a[i++];return s/c;
}

console.log(d([5,6,1]));
```

vs.

```JS
function durchschnitt(daten){
	anzahl = daten.length;
	index = 0;
	summe = 0;
	while(index < anzahl)
		summe = summe + daten[index++];
	return summe/anzahl;
}

console.log(durchschnitt([5,6,1]));
```

Letzteres deutlich lesbarer und verständlicher!

- zweckschreibende Namen
- Aussprechbare Namen
- keine "Ungarische Notation" (Typ im Namen)
- Keine Kennzeichnung von Instanzvariablen durch Präfixe
- Namenskonventionen beachten
- DEnglisch vermeiden

## Kurze Funktionen
- Funktionen sollten nur eine Aufgabe erfüllen (macht auch den Namen sprechender)
- Möglichst nur eine Einrückung in der Funktion
- Zerlegung in mehrere kleinere Funktionen

Warum?
- Verringerung kognitiver Belastung
- Bessere Wiederverwendbarkeit
- Implizite Dokumentationen

## SOLID - Prinzipien

Single Responsibility Principle
"Eine Klasse sollte nur einen Zweck verfolgen."

Open-Closed Principle
"Eine Klasse sollte erweitert werden können, ohne sie verändern zu müssen."

Liskov Substitution Principle
"Eine Instanz einer Unterklasse kann eine Instanz der Oberklasse ersetzen."

Interface Segregation Principle
"Die Fähigkeiten einer Klasse sollten über kleine Interfaces erreichbar sein."
	-> *Klassen sollten nur die Methoden implementieren, die sie wirklich brauchen*
Dependency Inversion Principle
"Programmieren gegen Interfaces, nicht gegen konkrete Implementierungen."
	-> *statt direkt mit Klasse zu arbeiten, sollte der Code mit Interfaces arbeiten*

> [!note]- Erklärung + Beispiel Liskov
> **Liskov Substitution Principle (LSP)**
> 
> **"Eine Instanz einer Unterklasse sollte eine Instanz der Oberklasse ersetzen können."**
> 
> Das bedeutet: Wenn du eine Unterklasse verwendest, sollte sie sich **wie ihre Oberklasse verhalten**. Der Code, der mit der Oberklasse arbeitet, sollte mit der Unterklasse **ohne Probleme** funktionieren, ohne Anpassungen vorzunehmen.
> 
> **Beispiel:**
> Angenommen, du hast eine Oberklasse `Bird` mit der Methode `fly()`:
> 
> ```java
> class Bird {
>     void fly() {
>         System.out.println("Flying...");
>     }
> }
> ```
> 
> Jetzt erstellst du eine Unterklasse `Penguin`. Aber Pinguine können nicht fliegen!  
> 
> Wenn du `Penguin` in einem Kontext verwendest, der erwartet, dass alle `Birds` fliegen können, bricht der Code:
> ```java
> class Penguin extends Bird {
>     @Override
>     void fly() {
>         throw new UnsupportedOperationException("Penguins can't fly!");
>     }
> }
> ```
> 
> Das verletzt das Liskov-Prinzip, weil ein `Penguin` **keinen gültigen Ersatz für einen `Bird`** darstellt.
> 
> **Bessere Lösung:**
> Passe die Hierarchie an, sodass nur flugfähige Vögel die `fly()`-Methode haben:
> ```java
> class Bird { }
> 
> class FlyingBird extends Bird {
>     void fly() {
>         System.out.println("Flying...");
>     }
> }
> 
> class Penguin extends Bird {
>     // Kein Fly mehr hier, weil Pinguine nicht fliegen
> }
> ```
> 
> Jetzt ist das Verhalten klar, und das Prinzip wird eingehalten.
