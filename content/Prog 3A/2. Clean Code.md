## Wiederholung
Was ist static?

```java
public static void main(String[] args) {

}
```

static -> Methode gehört zur Klasse 
-> Methode kann aufgerufen werden bevor es Objekte gibt (man braucht keine Objekte!). 

```java
public static void main (String [] args) {
 abc();
}

public void abc() { ... }
```
-> funktioniert nicht
- es wurde noch kein Objekt angelegt und somit kann die Methode nicht aufgerufen werden

Geht das?
```java
public static void main (String [] args) {
	Zahlenfilter z = new Zahlenfilter();
	z.xyz();
}

public void abc() { ... }

public void xyz() {
	abc();
}
```
Wir brauchen immer ein Objekt um eine nicht statische Methode aufzurufen!

```java
public static void main (String [] args) {
	Zahlenfilter z = new Zahlenfilter();
}

public void abc() {
	System.out.println("Hello");
}
```
System.out.println:
<u>System</u> -> Klasse von java.lang
<u>out</u> -> statische Klassenvariable / Ausgabestream

## Clean Code

Was können wir tun um menschenverständlichen Code zu schreiben:

### Aussagekräftige Namen

```JS
function d(a){
	c=a.length, i=0, s=0;
	while(i<c)s+=a[i++];return s/c;
}

console.log(d([5,6,1]));
```

vs.

```JS
function durchschnitt(daten){
	anzahl=daten.length;
	index=0;
	summe=0;
	while(index < anzahl)
		summe = summe + daten[index++];
	return summe/anzahl;
}

console.log(durchschnitt([5,6,1]));
```

Letzteres deutlich lesbarer und verständlicher!

- zweckschreibende Namen
- Aussprechbare Namen
- keine "Ungarische Notation" (Typ im Namen)
- Keine Kennzeichnung von Instanzvariablen durch Präfixe
- Namenskonventionen beachten
- DEnglisch vermeiden

## Kurze Funktionen
- Funktionen sollten nur eine Aufgabe erfüllen (macht auch den Namen sprechender)
- Möglichst nur eine Einrückung in der Funktion
- Zerlegung in mehrere kleinere Funktionen

Warum?
- Verringerung kognitiven Belastung
- Bessere Wiederverwendbarkeit
- Implizite Dokumentationen

## Step-Down-Regel
Code sollte wie eine Erzählung von oben nach unten gelesen werden.

## Vermeidung von Switch
Im Code verstreute Fallunterscheidungen sind bei jahrelanger Code-Wartung ein Alptraum.

Was passiert:
- wenn ein neuer Fall auftritt?
- wenn ein Fall wegfällt?

```java
public Money calculatePay(Empleyee e)
	throws InvalidEmployeeType {
	switch (e.type) {
		case COMMISSIONED:
		 return calculateCommissionedPay(e);

		case HOURLY:
			return calculateHourlyPay(e);

		case SALARIED:
			return calculateSalariedPay(e);

		default:
			throw new InvalidEmployeeType(e.type);
	}
}
```

Besser: Polymorphismus

Unterklassen überschreiben die Implementierung der Oberklasse.

Oberklasse:
```java
class Employee {
	Money calculatePay(){
	 // default Implementierung
	}
}
```

Unterklassen:
```java
class CommissionedEmployee
	extends Employee {

	Money calcualtePay()
		// Commissioned Implementierung
	}
```

```java
class HourlyEmployee
	extends Employee {
	
	}
```
...

Nutzung:
```java
Employee e = ...

Money m = e.calculatePay();
```

-> Wartungsfreundlicher

## SOLID - Prinzipien

Single Responsibility Principle
"Eine Klasse sollte nur einen Zweck verfolgen"

Open-Closed Principle
"Eine Klasse sollte erweitert werden können, ohne sie verändern zu müssen"

Liskov Substitution Principle
"Eine Instanz einer Unterklasse kann eine Instanz der Oberklasse ersetzen"

Interface Segregation Principle
"Die Fähigkeiten einer Klasse sollten über kleine Interfaces erreichbar sein"

Dependency Inversion Principle
"Programmieren gegen Interfaces"
