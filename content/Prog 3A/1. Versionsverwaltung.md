## Versionskontrolle

### Beispiel Softwareentwicklung
 - iterativer Prozess → man nähert sich dem Ergebnis schrittweise
 - wie springen wir bei einem Fehler in unserem Code zu einer früheren Version des Codes?

<u>Erste Lösungsidee</u>:
- Dateien einzeln sichern (auf bspw. Arbeitsplatz/lokal)
→ kompliziert (Benennung, Speicher)

<u>Besser</u>:
Versionsverwaltungssysteme

![](static/assets/proga-1.png)
- Code wird in Source Code Management System (SCM) übergeben
- Nur Änderungen werden gespeichert
	 → gut für Textdateien
	 → nicht für Bilder, Binärdateien

![](static/assets/proga-2.png)
- bestimmte Versionen rausziehen
	 → er nimmt den nächst-gelegenen kompletten Stand
	 → fügt Änderungen hinzu und gibt Code

![](static/assets/proga-3.png)
- können verschiedene Projekte organisieren
- für jeweilige Projekte erstellt er eigene Repositories

![](static/assets/proga-4.png)
- **Commit** schickt Arbeitsverzeichnis in unser Versionsverwaltungssystem (Repository)
- da nur Änderungen gespeichert werden → ist Speicherbedarf gering
- Ablagestruktur ist verborgen → es müssen keine Datennamen vergeben werden

Beispiel für so ein Versionsverwaltungssystem ist **Git**

## Git
![](static/assets/proga-5.png)
- Um die Daten zu sichern und ggf. auch anderen Entwicklern zugänglich machen zu können, sollte das Repository aber mit einem Server abgeglichen werden.
- Um Daten zu sichern & für andere verfügbar zu machen
	 → Repository mit Server abgleichen


```
git init (initialisiert ein leeres lokales Repository)
git add (fügt Daten zum Index hinzu)
git commit (überträgt die Dateien des Index ins lokale Repository)
git reset (setzt auf einen früheren Stand zurück)
git remote (verwaltet mögliche Remote-Repositories)
git push (überträgt vom lokalen zum Remote-Repository)
git clone (rekonstruiert das Remote-Repository auf einem Entwicklerplatz)
git fetch (holt die neuesten Änderungen vom Remote-Repository ins lokale Repository)
git merge (Integriert Änderungen in aktuellen Branch)
git pull (merge + fetch -> holt Änderungen aus einem Branch und integriert sie in lokalen Branch)
```

## Video: Git Grundlagen
- wir wollen den Quellcode in ein Repository übertragen
- dafür muss man erst ein Repository anlegen
- Kommandozeilenbefehl: `git init`
- in diesem versteckten git Verzeichnis werden die Deltas gespeichert
- nur mit git Befehlen können wir auf die Dateien in diesem Verzeichnis zugreifen → anders nicht lesbar/zugraifbar

Git übernimmt nicht automatisch alle Dateien die im Arbeitsverzeichnis liegen ins Repository
→ wenn Entwickler einen neuen "Stand" ins Repository übertragen will, müssen zwei Schritte durchlaufen werden:

1. Daten müssen in einen Index eingefügt werden → `add`
2. Index muss ins Repository eingefügt werden → `commit`

```
git add quellcode
git commit -m "Init"
```

Auf vorherigen Stand zurücksetzen:
```
git reset --hard HEAD
```

Bei mehreren Commits können alle bisherigen Commits so ausgelesen werden:
```
git log
```

Falls wir auf einen früheren Commit zurückgehen wollen:
```
git reset --hard HEAD^

oder

git reset --hard //gefolgt von Bezeichnung
```

## Video: Zentralisierter Git-Workflow

### Konflikte
- Entstehen, wenn in einer Datei von unterschiedlichen Entwicklern Änderungen vorgenommen und in ein zentrales Repository übertragen werden
- Git kann Konflikte lösen, wenn sie in der Datei an unterscheidbaren Stellen durchgeführt wurden
- Andernfalls sind die Konflikte manuell zu lösen

### Konflikte vermeiden
- Quelldateien klein halten → je kleiner die Datei, umso unwahrscheinlicher ist es, dass zwei Personen zeitgleich daran arbeiten
- Häufig mit `git pull`Änderungen vom zentralen Repository abrufen → je kürzer die Zeitspanne ist, umso eher kann Git diese automatisch zusammenführen
- Grobe Zuständigkeiten vereinbaren → aber es muss möglich sein, dass jeder alle Codeteile noch ändern kann ("Collective Ownership")
