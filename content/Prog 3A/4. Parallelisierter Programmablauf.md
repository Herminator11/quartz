## Threads
- erlauben einem Programm effizienter zu arbeiten, indem es mehrere Sachen auf einmal macht
- können genutzt werden um komplizierte Aufgaben im Hintergrund zu bearbeiten, ohne das Main-Programm zu unterbrechen

### Single-Thread vs. Multi-Thread
#### Single-Thread Programme
- läuft in einem einzigen Thread, der von der JVM zu Beginn des Programms erstellt und gestartet wird.
- innerhalb des Threads wird ein Befählszähler verwendet. Dieser zeigt immer auf die gerade auszuführende Anweisung.
- Linearer Ablauf -> jede Anweisung wird nacheinander abgearbeitet.
- Thread endet, wenn Befählszähler da Ende des Programms erreicht

#### Multi-Thread Programme
- kann mehrere Thread gleichzeitig ausführen
- jeder Thread im Multi-Thread hat eigenen Befehlszähler und Stack, aber teilen sich Heap und globale Variablen
	-> können auf gleiche Objekte zugreifen -> Synchronisationsproblem
- JVM verwaltet die Threads
	-> entweder abwechselnd auf einem Core oder gleichzeitig auf mehreren Cores
### Syntax
Es gibt zwei Wege einen Thread zu erstellen.

Es kann erstellt werden, indem man ein Thread erweitert und die `run()` Methode überschreibt:
```java
public class Main extends Thread {
	public void run() {
		System.out.println("This code is running a thread");
	}
}
```

Ein anderer Weg einen Thread zu erstellen is es das `Runnable`Interface zu implementieren:
```Java
public class Main implements Runnable {
	public void run() {
		System.out.println("This code is running a thread");
	}
}
```

Der Unterschied zwischen einen Thread erweitern and implementieren:
Der große Unterschied ist, dass wenn eine Klasse die Thread Klasse erweitert, du keine andere Klasse erweitern kannst. Aber wenn man das `Runnable`Interface implementiert, ist es möglich andere Klassen zu erweitern.
### Threadprogrammierung in Java
```java
public class Task1 implements Runnable { 
	public void run() { 
		long start = System.currentTimeMillis(); 
		while (System.currentTimeMillis() < start + 2000);
		System.out.println("Task 1 fertig"); 
	} 
}
```

> [!note]- Was passiert in diesem Code:
> 1. `Runnable` und die `Task1`-Klasse
> - **`Runnable`** ist eine Schnittstelle in Java, die eine Methode namens `run()` enthält. Klassen, die `Runnable` implementieren, müssen diese Methode definieren. `Runnable` wird oft verwendet, um Code zu schreiben, der in einem separaten Thread ausgeführt werden soll.
> - In diesem Beispiel implementiert die Klasse `Task1` das `Runnable`-Interface. Sie hat eine `run()`-Methode, die den Code enthält, den wir parallel (in einem separaten Thread) ausführen möchten.
> 
> 2. Was passiert in der `run()`-Methode von `Task1`?
> - Die `run()`-Methode startet eine Schleife, die zwei Sekunden (2000 Millisekunden) lang läuft. Das erreicht sie, indem sie den aktuellen Zeitpunkt (`System.currentTimeMillis()`) mit einem Startzeitpunkt vergleicht. Die Schleife wird so lange ausgeführt, bis die Differenz 2000 Millisekunden erreicht.
> - Nach Ablauf der zwei Sekunden wird die Nachricht `"Task 1 fertig"` auf der Konsole ausgegeben.
> 
> 3. Die `main`-Methode
> - In der `main`-Methode wird ein Objekt von `Task1` erstellt und als `Runnable` referenziert.
> - Dann wird ein neuer **Thread** erstellt, der diesen `Runnable`-Code (`Task1`) ausführt. Das passiert mit `new Thread(parallelerCode)`.
> - Der Thread wird durch `thread.start()` gestartet. Dies bedeutet, dass der `run()`-Code von `Task1` jetzt in einem separaten Thread ausgeführt wird, parallel zum Hauptprogramm (also zur `main`-Methode).
> - Direkt nach dem Starten des Threads gibt die `main`-Methode `"main Ende!"` aus, bevor der `run()`-Code von `Task1` beendet ist.

### Klasse Thread
- Java stellt die Klasse `Thread` bereit (teil der Standardbibliothek), um Threads zu erstellen und zu verwalten

```java
public static void main(String[] args) { 
	Runnable parallelerCode = new Task1(); 
	Thread thread = new Thread(parallelerCode); 
	thread.start(); 
	System.out.println("main Ende!"); 
}
```
- Klasse wird geschrieben, die `Runnable` implementiert
- neuer Thread wird mittels `Thread`-Klasse erweitert
- Ausführung des neuen `Threads` beginnt, sobald die Methode `start` aufgerufen wird.
### Busy Wait
- Thread bleibt in der Schleife bis Bedingung erfüllt ist.
- Rechenzeit wird trotzdem verbraucht, obwohl eigentlich nur darauf warten soll, dass Bedingung eintritt
```java
public class BusyWaitExample implements Runnable {
    public void run() {
        long start = System.currentTimeMillis();
        while (System.currentTimeMillis() < start + 2000) {
            // Nichts tun, nur warten
        }
        System.out.println("2 Sekunden sind vergangen!");
    }
}

public static void main(String[] args) {
    Thread thread = new Thread(new BusyWaitExample());
    thread.start();
}

```
-> Thread wartet zwei Sekunden, indem er immer wieder aktuelle Zeit überprüft.

Thread benutzt weiterhin CPU! -> Besser: `Thread.yield()`
### Busy Wait - etwas besser
**Thread.yield()**:
- teilt dem Scheduler mit, dass der aktuelle Thread gerade keine Rechenzeit benötigt.
- andere Threads können bearbeitet werden.

**Thread.sleep():**
- Thread wird für bestimmte Zeit in Schlafmodus versetzt
- blockiert keine CPU-Ressourcen

### Threadsynchronisation
- mehrere Threads in einem Programm koordinieren
**Join**:
- ein Thread möchte auch einen anderen Thread warten
- mit `join()` warten, bis anderer fertig mit Arbeit ist

**Race Conditions**:
- alle Threads eines Programms teilen sich gemeinsamen Adressraum
- gleichzeitiger Zugriff auf eine geteilte Ressource führt zu einer Race Condition
- Folge: Unvorhersehbares und inkorrektes Verhalten
- Lösung: Synchronisation oder Locks

### Locking
- mit `synchronized` kann ein kritischer Abschnitt definiert werden
	-> dieser darf immer nur von einem Thread betreten werden
- JVM macht eine Warteschlange aller Threads
	-> wenn einer betreten wird, schickt er alle anderen schlafen
**Nachteil:**
- Gefahr von Deadlocks

### Deadlock
- entsteht, wenn Threads mehrere kritische Abschnitte betreten wollen und die sich wechselseitig daran hindern.

## Stack memory vs. Heap memory
Welcher Thread sieht was?
- globale Variablen sehen alle Threads
- Instanzvariablen sehen alle Threads
- lokale Variablen landen auf den Stack und nur der Thread der ausführt, sieht diese

> [!info] Stack memory
> *...is used for thread execution, it stores local variables, method calls and arguments within one thread. Stack memory is small and fast. Stack memory is always referenced in LIFO (Last-In-First-Out) order.*

> [!info] Stack memory
> *...stores objects that are created and the objects takes up heap memory until there is a reference to them from somewhere in the application. As soon as there is no reference to an object, the object is removed by the GC and heap memory is freed up. Heap memory is big and slow.*
