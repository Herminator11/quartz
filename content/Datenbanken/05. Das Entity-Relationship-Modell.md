## Das ER-Modell
- Darstellung von Daten in unserem DBS
- Speichern von den Daten
- zeigt uns die Beziehung von den Daten zueinander

- es gibt Entitäten, Attribute, Beziehungen

### Entität (Entity)
- sind Vorgänge/Objekte/Vorgänge aus der Realität (z.B. Kunde)
- werden als Rechteck dargestellt

### Attribute & Schlüssel
- beschreiben die Entitäten -> geben ihnen Eigenschaften
- werden als Ovale gekennzeichnet
- z.B. hat eine Person: Alter, Name oder Beruf als "Eigenschaften"
- einzelne Attribute können <span style="background:#ff4d4f">eindeutig identifiziert</span> werden 
	-> dann heißen sie <span style="background:#ff4d4f">Schlüssel</span>

#### Primärschlüssel
- identifiziert jede Entity eindeutig
- z.B. KundenNr. -> ist einzigartig und darf nie doppelt oder leer sein
- wird unterstrichen

#### Fremdschlüssel
- verweist auf Primärschlüssel einer anderen Entity
- z.B. bei einer Bestellung ist die KundenNr. da -> diese zeigt eine Verbindung zum Kunden
- werden gestrichelt unterstrichen

### Beziehungen
- als Raute dargestellt
- verdeutlichen die Verbindung zwischen zwei Entities
- ebenso wird daneben die Kardinalität angegeben

<u>Kardinalität</u>:
- "wie viele Instanzen einer Entity mit wie viele Instanzen einer anderen Entity können verknüpft werden"

- Es gibt drei Beziehungsarten:
	- 1:1 - Beziehung
	- 1:n - Beziehung
	- n:m - Beziehung

#### 1:1 - Beziehung
- Entities treten nur einmal in der Beziehung auf
- z.B. jeder Mitarbeiter besitzt nur einen Mitarbeiterausweis und jeder Mitarbeiterausweis ist eindeutig einem Mitarbeiter zugeordnet

#### 1:n - Beziehung
- eine Entity ist mit einer anderen verknüpft
- aber andersherum kann sie mit mehreren Entities verknüpft sein
- z.B. eine Bestellung kann immer nur von einem Kunden aufgegeben werden
	- ein Kunde kann aber mehrere Bestellungen haben

#### n:m - Beziehung
- eine Entity kann mehrere Instanzen haben und andersherum
- z.B. ein Lieferant kann mehrere Produkte liefern und ein Produkt kann von mehreren Lieferanten geliefert werden

## Allgemeine Funktionen
- NVL, NVL2, NULLIF, COALESCE

### NVL
- ersetzt einen `NULL`-Wert durch einen angegebenen Wert

```sql
NVL(expression, replacement_value)
```
- können wir nutzen, wenn wir `NULL`-Werte durch etwas **sinnvolles** ersetzen wollen
z.B.:
```sql
SELEC Nachname, NVL(TO_CHAR(Vorgesetzter),
				   'kein Vorgesetzer')
```
- ein "Fallback-Wert"

### NVL2
`NVL2 (expression, value_if_not_null, value_if_null)`
- wenn nicht `NULL` liefert `value_if_not_null`
- wenn `NULL`, liefert `value_if_null`

Beispiel:

```sql
SELECT Name,
	NVL2(Vorgesetzter, 'hat Vorgesetzten', 'kein Vorgesetzter')
FROM Mitarbeiter;
```
- Gibt aus, ob ein Mitarbeiter einen Vorgesetzten hat oder nicht.

### NULLIF
- vergleicht zwei Ausdrücke. Wenn beide gleich sind, gibt es `NULL` zurück.
- Andernfalls gibt es den ersten Ausdruck zurück.
`NULLIF(expression1, expression2)`
- nützlich, um doppelte Werte zu vermeiden

Beispiel:
```sql
SELECT Name,
	NULLIF(Gehalatsbonus, 0) AS Bonus
FROM Mitarbeiter;
```
- wenn Bonus `0` ist, wird `NULL` zurückgegeben. Sonst wird der Bonus ausgegeben.

### COALESCE
- gibt den ersten Nicht-NULL-Werte aus einer Liste von Ausdrücken zurück
`COALESCE(expression1, expression2, …, expressionN)`
- Ideal für mehrere Fallback-Werte

Beispiel:

```sql
SELECT Name,
	COALESCE(Telefonnummer, Mobilnummer, 'Keine Nummer') AS Kontakt
FROM Mitarbeiter;
```
- Gibt die Telefonnummer zurück, oder die Mobilnummer, oder 'Keine Nummer', falls beide `NULL` sind.

> [!summary]- Zusammengefasst
> - **NVL:** Ersetzt `NULL` mit einem bestimmten Wert.
> - **NVL2:** Unterschiedliche Werte für `NULL` und `Nicht-NULL`.
> - **NULLIF:** Gibt `NULL`, wenn zwei Werte gleich sind.
> - **COALESCE:** Erster Nicht-NULL-Wert aus einer Liste.

## CASE-Expressions
- prüft verschiedene Bedingungen 

Syntax:

```sql
CASE expression
	WHEN value1 THEN result1
	WHEN value2 THEN result2
	...
	ELSE defaul_result
END
```
Beispiel für eine Searched CASE-Expression:
- verwendet, um komplexe Bedingungen auszuwerten.

```sql
SELECT Name,
	CASE
	WHEN Gehalt > 5000 THEN 'Hoher Verdienst'
	WHEN Gehalt BETWEEN 3000 AND 5000 THEN 'Mittel'
	ELSE 'Niedrig'
END AS Gehaltskategorie
FROM Mitarbeiter;
```
